# Lotto Lab — 開發覆盤（v1.2 建置）

> 給洛伊與 fable5 討論用。這份文件回顧「從一份凍結規格到上線實驗」的整個過程、
> 關鍵決策、目前結果，以及最後一節專門討論「開放給更多人一起玩」的選項與科學界線。

## 1. 一句話

依 `lotto-lab-spec-v1.2.md`（凍結規格）從零建置一台「證明或推翻大樂透隨機性」的實驗裝置，
M1–M7 全數完成、上線、並於 2026-07-19 完成預先註冊（啟動期 115000072）。

## 2. 最終成果

- **資料層**：台彩官方 JSON API（2007+，約 2,153 期，全 `full`）＋ 北銀時代 2004–2006 延伸層
  （312 期，雙第三方對帳、`numbers_only`）。
- **引擎**：`Game` 抽象（不寫死 49 選 6，為未來威力彩/今彩539 預留）、walk-forward + 前視保護、
  雙軌 ROI（理論軌為推論主指標、實際軌展示用，加碼獎項兩軌均不計入）。
- **策略庫**：A–F 共 68 個（A50 對照組 + B 統計 5 + C 序列 3 + D 養牌 3 + E 包牌 5 + F 反熱門 2）。
- **統計**：permutation test + Benjamini–Hochberg FDR、對照組冠軍比較；硬體隨機性審計
  （單號/特別號/配對共現/分段卡方）。合成資料驗收（植入偏差被偵測、純隨機不誤報）。
- **私人加密層**：WebCrypto AES-256-GCM + PBKDF2 600k；`engine.js`（過共用測試向量）、
  `personal.html`（解密計分 + 網頁編輯器）、`add_bet.py`（CLI，與瀏覽器互通）。
- **展示 + 自動化**：公開 dashboard（GitHub Pages）＋ 每日冪等 GitHub Actions。
- **測試**：44 個（含前視保護、手工對照、合成統計驗收、密文無明文檢查、跨語言互通）。

## 3. 開發歷程（本 session）

1. 讀凍結規格 → 定計畫（先驗證外部依賴，OK 就建核心，再逐模組）。
2. **外部依賴驗證**：發現台彩有乾淨 JSON API（免爬 HTML），風險消除。
3. 建核心 M1+M2+A → 全管線煙霧測試跑通 2,153 期無崩潰。
4. 逐步補 M3 B–F → M4 統計（先過合成資料）→ M5 加密層（瀏覽器實測 add_bet.py 寫的密文由
   WebCrypto 解回）→ M6 dashboard → M7 workflow。
5. 部署：repo 轉公開、GitHub Pages 上線。
6. **加碼決策**：洛伊提出納入 2004–2006 北銀資料 → 定為「延伸視圖 + 雙第三方對帳 + 主結論維持官方源」。
7. **預先註冊**：凍結 config（啟動期 115000072），commit 時間戳 = 註冊時點。
8. **易用性**：私人層加網頁編輯器（免終端機）、寫使用說明與本覆盤。

## 4. 關鍵決策與理由（易錯區留痕，詳見 DECISIONS.md）

| 決策 | 選擇 | 為什麼 |
|---|---|---|
| ROI 雙軌 | 理論軌為推論主指標 | 浮動獎金受銷售熱潮/累積頭獎影響，會讓對照組運氣雲出現極端離群、扭曲顯著性門檻——那是在測「銷售波動」不是「預測力」 |
| 加碼獎項 | 兩軌均不計入 | 大紅包「完全對中」制對所有號碼均勻→零鑑別力；單次撞中百萬毛刺會摧毀運氣雲比較 |
| permutation null | 同結構隨機重抽（single/wheel7/wheel8，依 bet_type 快取） | 保留各策略下注量結構，直接檢定「號碼選擇是否超越同量隨機」 |
| 北銀 2004–2006 | 延伸視圖、雙第三方對帳、numbers_only、不進策略推論 | 無官方源→信任層降級但對帳紀律不降；主結論站最高信任層，就算第三方被質疑也只傷延伸視圖 |
| 私人層 | 純 client-side WebCrypto，不用 GitHub Secrets | Actions log/產物有外洩面；client-side 攻擊面最小，密文可安心放公開 repo |
| 玩法擴充 | Game 插件抽象、v1 只實作 lotto649 | 核心資產是「防自欺機器」，與玩法無關；複製多份等於讓最易錯的統計核心各自漂移 |

## 5. 目前科學結果（觀察段，非最終結論）

- **硬體審計**（全歷史 2,153 期）：單號卡方 p=0.93、配對共現 p=0.99（符合均勻隨機）；
  特別號 p=0.044（多重檢定中單一項的邊際值，不構成警訊）。延伸視圖（含北銀，2,465 期）
  單號 p=0.92；2004–2006 各年分段 p=0.58–0.95，未見偏差集中於承辦商更換邊界。
- **策略驗證**（觀察性、全歷史）：FDR 校正後 **0/68 策略**顯著優於對照組冠軍 → 符合 H0。
- ⚠️ 以上為歷史觀察；**正式結論以前瞻段跑滿約 200 期後的報告為準**（結果甲/乙見規格 §5.3）。

## 6. 給 fable5 討論：開放給更多人一起玩

既然 repo 公開，讓更多人參與是自然的想法。但這裡有一條**不能踩的科學界線**，先講清楚，
再列可行選項。

### ⛔ 不能做的：把使用者策略加進「當前這個已註冊的實驗」
預先註冊的價值在於「策略清單在看到結果前就凍結」。中途加入外部策略會：
① 破壞凍結（等於事後改規則）；② 改變 FDR 的 N（多重比較校正基準漂移）；
③ 引入「多人狂試 → 總有人隨機中」的 p-hacking 風險。**這條線守住，整個實驗才有公信力。**

### ✅ 選項 A：Playground（沙盒）——推薦優先做
一個獨立頁面，讓任何人**用自己的選號邏輯，對歷史資料即時回測**、疊到運氣雲上自己看。
- 特性：純觀察、client-side、不寫回任何官方結果、不參與 FDR。
- 好處：好玩、教育性強、零科學風險；技術上幾乎是現成的（engine.js + 運氣雲已具備）。
- 成本：低。等於把「運氣雲 + 一個號碼產生器輸入框」開放出來。

### ✅ 選項 B：第二梯次的獨立預先註冊（社群實驗）
想讓大家的策略「正式參賽」，就開一個**全新的預先註冊梯次**：自己的策略清單、自己的對照組、
自己的啟動期與停止規則，與現行大樂透實驗**不混池檢定**（這正是規格 §10「每個新玩法/新梯次
= 一次獨立預先註冊」的精神延伸）。
- 特性：科學上乾淨；報名截止 = 凍結，之後才開跑。
- 成本：中。需要收集/凍結策略、防重複、防前視（使用者策略也要走引擎的 walk-forward 保護）。

### 🔒 選項 C：多人真實投注（crowd）——需重新檢視威脅模型
若要讓多人記錄各自的真實投注，現行私人層的威脅模型（單人、單密語）要重新設計
（每人一把金鑰？資料放哪？隱私邊界？）。建議**最後再議**，且與科學實驗解耦。

### 建議路線
先做 **A（沙盒）**，好玩又零風險；有社群動能再規劃 **B（第二梯次）**；**C** 最後再談。
關鍵原則不變：**任何開放都不得污染已註冊實驗的主結論。**

## 7. 已知限制 / 未來工作

- 北銀延伸層 period 為重建 ROC 序（非官方原始期號）；prizes/sales 缺（numbers_only）。
- F 組（反熱門）理論優勢只在高額分獎體現，低獎項回測量測不到，靠 6.3 間接驗證前提。
- 200 期檢定力只足以偵測「大幅」偏離；接受 H0 ≠ 證明完美隨機。
- 未來玩法（威力彩/今彩539/樂合彩/星彩）：Game 插件已預留，各需獨立預先註冊。
