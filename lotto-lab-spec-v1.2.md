# Lotto Lab — 大樂透隨機性驗證實驗裝置 規格書 v1.2

> 專案定位：這不是報明牌系統，而是一個「證明（或推翻）隨機性」的實驗裝置。
> 預設假說 H0：所有策略的長期每元投報率，統計上無法與純隨機對照組區分。
> 前瞻實驗期：約兩年（~200 期）。
>
> **v1.1 變更摘要**：① 統計拆分為「策略驗證」與「硬體隨機性審計」兩獨立模組；② ROI 雙軌制（理論期望值軌為推論主指標）；③ 熱門度分析以銷售量標準化；④ data_quality 資料品質分級；⑤ 新增私人加密投注層（洛伊實際投注追蹤，號碼永不以明文出現在公開 repo）；⑥ 公開策略 D01 由凍結隨機養牌取代，夢境號碼移入私人層。
>
> **v1.2 變更摘要**：⑦ 支援節慶加碼活動（春節／端午／中秋「百萬大紅包」）：排程改為每日冪等執行以因應春節連續開獎；加碼獎號以帶格式標記的原始結構入庫（§2.3）；**核心實驗兩軌均不計入加碼獎項**（零策略鑑別力＋高運氣雲污染，理由見 §4.2）；私人層實際投注**必須**比對加碼獎號（§7.3）。⑧ 新增多玩法擴充架構原則（§10）：玩法抽象為插件介面，v1 僅實作大樂透。
>
> **給實作端（Opus 主控＋Sonnet 子代理）**：本文件為凍結規格。所有架構與統計決策已在規格層完成，實作端不需要、也不應該重新裁決設計問題。遇到規格未覆蓋的情況：小於模組邊界的實作細節由 Opus 裁決並記錄於 DECISIONS.md；涉及統計方法或資料語意的變更一律回報洛伊，不得自行變更。

## 2. 資料層（M1）

### 2.1 draws.json Schema

```json
{
  "meta": { "game": "lotto649", "updated": "ISO8601", "sources": ["taiwanlottery", "backup"] },
  "draws": [
    {
      "period": "113000001",
      "date": "2024-01-02",
      "numbers": [3, 12, 19, 27, 33, 41],
      "special": 7,
      "sales_amount": 512345600,
      "prizes": {
        "t1": { "winners": 0, "amount": 0 },
        "t2": { "winners": 1, "amount": 12345678 },
        "t3": { "winners": 15, "amount": 150000 },
        "t4": { "winners": 40, "amount": 30000 },
        "t5": { "winners": 900, "amount": 2000 },
        "t6": { "winners": 1200, "amount": 1000 },
        "t7": { "winners": 20000, "amount": 400 },
        "t8": { "winners": 30000, "amount": 400 }
      },
      "data_quality": "full",
      "promo": null
    }
  ]
}
```

- `numbers` 升冪排序。`sales_amount` 為當期銷售金額（新增欄位，供 6.3 熱門度標準化；÷50 得總注數估計）
- `data_quality` 三級：
  - `full`：號碼＋完整 prizes＋sales_amount
  - `partial`：號碼齊全，prizes 或 sales 缺漏 → 引擎自動改用理論獎額計分（見 4.2），熱門度分析跳過該期
  - `numbers_only`：僅有號碼 → 只參與命中統計與理論軌計分
- 早期期數（2004–2008 前後）預期大量 `partial` / `numbers_only`，屬正常現象，引擎不得因欄位缺漏崩潰或回傳零值
- 雙源比對：主源台彩官網，備源第三方資料站。**號碼不一致 → 標記 conflict、停止 commit、開 issue 等人工裁決**；僅獎金細節不一致 → 以官網為準並記 log

### 2.3 節慶加碼活動資料（v1.2 新增）

台彩近年於春節、端午、中秋推出「百萬大紅包」類加碼，**格式逐年變動**，已知至少兩種：

- **sets 格式**（近年端午／中秋、早期春節）：例行開獎外，另開出多組（如 100 組）各 6 碼的促銷獎號；投注號碼「完全對中」任一組得 100 萬（多注均分）；未送完組數延續至下期；送完或活動截止即停
- **pool 格式**（2026 春節）：每期加開 1 組 9 碼獎號池＋1 個小紅包獎號；對中池中任 6 碼得大紅包（最高 100 萬）；對中任 5 碼＋小紅包獎號得 10 萬；設有總組數上限（如大紅包 480 組），提前送完則活動提前結束

另注意：**春節加碼期間大樂透連續每日開獎**（如 2026 年 2/12–3/3 連續 20 天），打破週二五節奏。

`promo` 欄位規格（無加碼時為 null）：

```json
"promo": {
  "format": "sets",              // "sets" | "pool" | 未來新格式
  "campaign": "2026-dragonboat", // 活動識別碼
  "data": {
    // sets:  { "sets": [[6碼], …], "prize": 1000000, "results": [各組中獎注數] }
    // pool:  { "pool": [9碼], "bonus": 小紅包號, "prizes": {"big": …, "small": …}, "results": {…} }
  }
}
```

- 設計原則：**原始結構入庫、格式標記分派**。爬蟲遇到無法識別的新格式時，存 `format: "unknown"` 附原始文字、開 issue，不阻斷例行開獎資料的 commit
- 加碼資料的消費者只有兩個：私人層實際投注比對（§7.3）與展示層的資訊性標註。核心實驗引擎（M2–M4）**不讀取此欄位**

**M1 驗收**：① 初始化爬完全歷史並產出 data_quality 分布報告；② 隨機抽 20 期與官網人工比對全對；③ 雙源衝突模擬測試觸發停止機制；④ 以 2026 春節（pool）與 2026 端午（sets）各抽一期驗證 promo 欄位解析正確；⑤ 春節連續開獎日期段的期數完整無缺漏。

## 3. 策略層（M3）

### 3.1 統一介面（與 v1.0 相同）

```python
class Strategy(Protocol):
    id: str            # 註冊後凍結
    group: str
    registered_period: str
    def predict(self, history: list[Draw]) -> list[Ticket]: ...
```

- 每注 NT$50。嚴禁前視：引擎層強制切片。參數註冊即凍結；要改開新 id，舊策略不刪（防倖存者偏差）
- **明確不做**動態權重／流年冠軍加碼機制（過擬合溫床）

### 3.2 策略清單（v1.1 註冊）

**A 組｜對照基準**：A00–A49，50 個 seeded 純隨機策略（seed 寫死在 config）

**B 組｜統計類**：B01 熱門號（近 50 期最高頻 6 號）、B02 冷門號、B03 遺漏回補、B04 尾數分布加權、B05 奇偶大小平衡

**C 組｜序列類**：C01 馬可夫共現轉移、C02 上期重複 1–2 號＋隨機、C03 差值分布生成

**D 組｜養牌類（公開）**：D01–D03，三組系統隨機產生後**凍結**的固定號碼（v1.0 的 D02–D04 升格；洛伊的夢境號碼不在公開層，見 §7）
- 驗證命題：固定號碼 vs 每期換號，長期績效無差異

**E 組｜包牌類**：E01 隨機 7 號包牌（7 注/期）、E02 隨機 8 號包牌（28 注/期）、E03 熱門 8 號包牌、E04 冷門 8 號包牌、E05 凍結 8 號包牌
- 驗證命題：包牌不改變每元期望值，只改變波動性與命中頻率

**F 組｜反熱門類**：F01 全大號（6 號皆 >31）、F02 反模式（排除順子／等差／同尾過多後隨機）
- 誠實標註：理論優勢僅在高額獎項分獎時體現，回測低獎項損益量測不到；由 6.3 間接驗證其前提

**M3 驗收**：每策略附單元測試；同一 history 輸入重跑兩次輸出必須相同（決定性測試，含 seeded 隨機）。

## 4. 回測引擎（M2）

### 4.1 Walk-Forward（與 v1.0 相同）

predict 只拿得到第 N 期以前的資料。**前視保護必須有專屬測試：餵入含未來期數的切片應丟例外。**

### 4.2 雙軌 ROI

- **理論軌 ROI（推論主指標）**：payout 以 §2.2 理論軌獎額計算。所有統計檢定（M4）只用理論軌。理由：浮動獎金受銷售熱潮與累積頭獎影響，會讓對照組運氣雲出現極端離群值、扭曲顯著性門檻——那是在測「歷史上的台彩銷售波動」而不是策略預測力
- **實際軌 ROI（展示用）**：payout 用當期實際公告獎額（data_quality=full 才計入實際軌；partial/numbers_only 期數實際軌標記為缺值）。呈現「若真金白銀跟單」的命運曲線
- 每策略每期記錄：cost、理論 payout、實際 payout、各獎項命中數
- **加碼獎項（§2.3）不計入任何一軌**（v1.2 凍結決策，實作端不得變更）。理由：① 大紅包為「完全對中」制，中獎機率對所有號碼組合均勻，對策略比較的鑑別力為零；② 全系統約 180 注/期 × 兩年約 60 個加碼期 × 每期近百組獎號，累積約有個位數百分比的機率讓某策略隨機撞中一次 100 萬，單一毛刺即摧毀運氣雲的顯著性比較。零鑑別力＋高污染 → 排除

### 4.3 評估指標

理論軌累積 ROI（主）、各獎項命中分布、累積損益曲線、最大回落、每元報酬標準差（包牌 vs 單注的波動性差異在此呈現）。

### 4.4 歷史段 vs 前瞻段

歷史段（2004–啟動日）一次跑完，僅供觀察；一切證據以前瞻段（啟動日後 ~200 期）為準。

**M2 驗收**：① 前視保護測試；② 手工計算 3 期 × 3 策略的對照樣本與引擎輸出全等；③ Python 引擎與 JS 移植版（M5 用）以共用測試向量驗證輸出一致。

## 5. 統計模組（M4）——兩個獨立子模組

### 5.1 策略驗證（前瞻段，200 期）

1. **對照組冠軍比較**：真策略需顯著贏過 A 組 50 個假算法的**最佳者**（不是平均）
2. **Permutation test**：候選策略 ROI 在隨機重抽分布中的百分位；p 值做 FDR 校正，N = 全部註冊策略數（含後續新增）
3. 誠實標註於報告：200 期的檢定力只足以偵測「大幅」偏離；接受 H0 ≠ 證明完美隨機，只代表偏差（若存在）小於本實驗可偵測門檻

### 5.2 硬體隨機性審計（全歷史 2,300 期，獨立於策略驗證）

- 單號頻率卡方適合度檢定：~2,300 期 × 6 球 ≈ 13,800 次抽球，每號期望 ~282 次，對數個百分點的系統性偏差有實用檢定力
- 特別號獨立做一次同樣檢定
- 號碼配對共現頻率檢定（成對出現是否偏離超幾何期望）
- 分段檢定：依年份分窗重跑，觀察偏差是否集中於特定時期（球組輪換的間接偵測）
- **明確不做 NIST 隨機性套件**：該套件為 RNG 位元流設計，樂透開獎的位元流映射勉強且樣本量遠不足，僅有儀式價值
- 誠實標註：台彩定期輪換球組與搖獎機，全歷史混池檢定只能偵測長期系統性偏差，無法偵測單一球組的短期物理瑕疵

### 5.3 停止規則與成功判準（預先註冊，本文件 commit 即為註冊時間戳）

- 跑滿 ~200 期出具最終報告，不提前下結論、不延長
- 結果甲（預期）：全策略理論軌 ROI 收斂於獎金返還率附近，與對照組冠軍無顯著差異 → 成功證明隨機性（在可偵測門檻內）
- 結果乙（異常）：某策略經 FDR 校正後仍顯著優於對照組冠軍 → 依序排查：程式 bug → 資料源污染 → 才考慮物理偏差
- **真錢防火牆**：系統只做虛擬損益；洛伊實際下注的金額與策略於場外決定，實際投注記錄（§7）僅供分析，不回饋進任何策略評分

**M4 驗收**：permutation test 以已知答案的合成資料驗證（植入人工偏差應被偵測、純隨機資料應不被誤報）。

## 6. 公開展示層（M6）

- 6.1 運氣雲：A 組 50 條理論軌曲線淡灰背景，真策略疊上；一眼看出誰跳出雜訊
- 6.2 組別對比：養牌組內部、包牌每元報酬、策略 vs 對照組分位數；理論軌／實際軌切換開關
- 6.3 熱門度分析：各期低獎項（t5–t8）中獎注數 **÷ 當期總注數（sales_amount/50）** = 每萬注中獎率。標準化後的異常升高才代表該期號碼是群眾偏好的熱門組合。僅用 data_quality=full 期數。誠實標註：電腦選號（均勻隨機）佔實際投注相當比例，會稀釋人為偏好訊號——本分析測到的是「剩餘人為偏好強度」，此稀釋程度本身即為有趣的觀察量
- 6.4 硬體審計頁：卡方結果、各號碼頻率偏差圖、分段窗檢定
- 6.5 最終報告頁：期滿自動生成

## 7. 私人加密投注層（M5）——v1.1 新增

### 7.1 目的與威脅模型

洛伊每期實際購買 2–3 組號碼（電腦選號隨機組／夢境號碼／祖父塔位號碼養牌）。需求：在同一網站上追蹤實際投注的投報率、與公開實驗做理論值 vs 實際值交叉分析，且**號碼本身在 repo 公開、網站爆紅、朋友檢視原始碼的情境下均不可見**。

威脅模型：防「任何能讀公開 repo 與網站原始碼的人」。不防：拿到洛伊密語的人、洛伊自己的裝置被入侵。此強度對本用途足夠。

**設計鐵律：私人號碼永不以明文出現在 repo、Actions log、或任何公開產物中。** 前端「密碼欄位」不是存取閘門而是解密金鑰——純靜態網站上任何非密碼學的密碼驗證都是裝飾品。

### 7.2 機制

- `data/private/bets.enc`：AES-256-GCM 密文，金鑰由通關密語經 PBKDF2（≥600k iterations, 隨機 salt 存於密文檔頭）導出。密文可安心放公開 repo
- 明文結構（僅存在於洛伊本機與瀏覽器記憶體）：

```json
{
  "recurring": [
    { "label": "夢境號碼", "numbers": [/*…*/], "since": "period" },
    { "label": "祖父塔位", "numbers": [/*…*/], "since": "period" }
  ],
  "bets": [
    { "period": "113000xx", "tickets": [
      { "label": "電腦選號", "numbers": [/*…*/] },
      { "label": "夢境號碼" },
      { "label": "祖父塔位" }
    ], "cost": 150 }
  ]
}
```

- `personal.html`：輸入密語 → WebCrypto 解密 → 以 `engine.js`（與 Python 引擎共用測試向量的移植版）在瀏覽器本地計分。解密失敗只顯示「無法解密」，不洩漏任何結構資訊
- `tools/add_bet.py`：本地記錄工具。輸入期別＋當期號碼＋密語 → 解密、附加、重加密、覆寫 bets.enc → 洛伊自行 commit。固定組（夢境／塔位）以 label 引用不重複輸入
- GitHub Actions **不接觸**密語與明文（不用 Secrets 方案：Actions log 與 results 產物有外洩面，client-side 方案攻擊面最小）

### 7.3 私人分析頁功能

- 實際投注累積損益（理論軌＋實際軌雙曲線）vs「每期只買一注電腦選號」基準 vs 公開對照組分位數
- 三組號碼各自拆開的績效與命中分布：電腦選號 vs 夢境 vs 塔位——洛伊自己的養牌實驗，樣本雖小仍完整記錄
- 理論期望 vs 實際結果差異（運氣餘額：至今比數學期望多賺／多虧多少）
- 若某期漏買，該期標記缺席，不內插
- **加碼獎號比對（v1.2 新增）**：與公開實驗相反，私人層**必須**對加碼獎項計分——洛伊的真實彩券在加碼期間確實具備紅包對獎資格，那是真實的潛在損益。engine.js 依 `promo.format` 分派比對邏輯（sets：逐組完全比對；pool：計算對中池內碼數＋小紅包）。私人實際軌含加碼、理論軌不含（維持與公開實驗理論軌可比性），兩者差額即「加碼活動帶來的額外期望」單獨呈現

**M5 驗收**：① repo 全文檢索與建置產物檢索確認無明文號碼（自動化測試，grep 已知測試號碼型樣）；② 錯誤密語不洩漏資訊；③ engine.js 與 Python 引擎共用測試向量輸出全等；④ add_bet.py 往返（加密→解密）無損測試。

## 8. 自動化（M7）

```yaml
schedule: cron '0 15 * * *'     # 每日 UTC 15:00 = 台北 23:00（v1.2：因應春節連續開獎改為每日）
workflow_dispatch: {}            # 手動觸發
```

流程：爬蟲檢查是否有新開獎 → **無新期數則冪等退出（no-op，不 commit）** → 有新期數：雙源比對 → 更新 draws.json（含 promo 欄位）→ 全策略回測（理論＋實際雙軌）→ 更新 results/ → commit。任一步失敗開 issue、不 commit 半套資料。私人層不在此流程內（完全 client-side）。

冪等設計取代「開獎日曆」規則：不維護哪些日子有加開的清單（春節每日連開、活動提前結束等變因太多），每天檢查一次、有就處理，是最穩健的解法。

## 10. 多玩法擴充架構原則（v1.2 新增）

**決策：統一架構、單一 repo、單一 dashboard（玩法分頁），v1 僅實作大樂透。**

理由：本專案的核心資產是防自欺機器（對照組、walk-forward、permutation test、預先註冊），它與玩法無關。每個玩法各建一組程式等於把最易出錯的統計核心複製多份各自漂移（反熵原則反例）。

**v1 的義務**：M2 引擎在設計時即以 `Game` 抽象介面隔離玩法規則——號碼空間、抽出結構、注的格式、獎金表、計分函數皆由 Game 插件定義；引擎、統計模組、對照組機制、dashboard 框架不得寫死 49 選 6 的假設。此抽象在 v1 只有一個實作（lotto649），成本極低；事後重構的成本則極高。

**v1 明確不做**：其他玩法的爬蟲、策略、頁面。未來擴充參考：

| 玩法 | 結構差異 | 擴充成本評估 |
|---|---|---|
| 威力彩 | 38 選 6＋第二區 8 選 1，週一四開獎 | 低（新 Game 插件＋爬蟲） |
| 今彩539 | 39 選 5，每日開獎 | 低 |
| 樂合彩 | 直接沿用主遊戲開獎號碼，選 2–5 碼對獎 | 極低（資料層免費，只需計分規則） |
| 3星彩／4星彩 | 數字型（含位置），固定獎金，每日開獎 | 中（注結構不同）。註：固定獎金制無均分效應，反熱門邏輯（F 組）不存在，期望值完全固定——會是最快收斂的驗證 |

每個新玩法上線 = 一次獨立的預先註冊（自己的策略清單、對照組、停止規則），不與大樂透實驗混池檢定。
